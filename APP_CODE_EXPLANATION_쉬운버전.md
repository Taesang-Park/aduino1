## 1. 앱이 하는 일

### 1.1 무엇을 만들었나요?

여러분은 **스마트폰으로 물 마신 양을 자동으로 기록하는 앱**을 만들었어요!

```
🏗️ 하드웨어 부분 (Arduino):
   └─ 컵 받침대에 무게 센서가 있어요
   └─ 물이 든 컵을 올려놓으면 무게를 측정해요
   └─ 블루투스로 스마트폰에 정보를 보내요

📱 앱 부분 (Android):
   └─ 블루투스로 무게 정보를 받아요
   └─ 물을 마시면 자동으로 기록해요
   └─ 오늘 얼마나 마셨는지 보여줘요
   └─ 목표(2000ml)까지 얼마나 남았는지 알려줘요
```

### 1.2 실생활 비유로 이해하기

이 앱을 **학교 수업 관리 시스템**에 비유해볼게요:

| 실제 앱 | 학교에 비유하면 |
|--------|----------------|
| Arduino (무게 센서) | 출석 체크하는 선생님 |
| 블루투스 | 학교 방송 시스템 |
| 앱 (Android) | 학생부 기록부 |
| 데이터베이스 | 보관용 서류철 |
| 화면 (UI) | 게시판 |

## 2. 앱의 구조 (건물에 비유)

### 2.1 3층 건물로 이해하기

우리 앱은 **3층짜리 건물**처럼 만들어졌어요. 각 층마다 하는 일이 달라요!

```
┌─────────────────────────────────────┐
│     3층: 화면 (Presentation)         │   👤 사람들이 보고 만지는 곳
│     - 버튼, 글자, 그림               │
│     - 사용자가 보는 화면              │
└─────────────────────────────────────┘
           ↕️ 정보 전달
┌─────────────────────────────────────┐
│     2층: 계산 (Domain)               │   🧮 계산하고 판단하는 곳
│     - 목표 달성률 계산               │
│     - 남은 양 계산                   │
└─────────────────────────────────────┘
           ↕️ 정보 전달
┌─────────────────────────────────────┐
│     1층: 저장소 (Data)               │   📦 정보를 보관하는 곳
│     - 데이터베이스                   │
│     - 블루투스 통신                  │
└─────────────────────────────────────┘
```

**왜 이렇게 나눴을까요?**
- 마치 학교에서 학생부(기록), 교무실(계산), 게시판(표시)이 나뉘어 있는 것처럼,
- 각 층이 자기 일만 집중하면 관리가 쉬워져요!
- 한 층에서 문제가 생기면 그 층만 고치면 돼요!

### 2.2 파일 구조 (폴더 정리)

여러분의 방을 정리하는 것처럼, 코드도 폴더로 정리했어요:

```
📁 aduino1 (앱 이름)
  │
  ├─ 📁 data (1층 - 저장 관련)
  │   ├─ 📄 WaterRecord.kt        → "물 마신 기록" 정보
  │   ├─ 📄 WaterDao.kt           → "데이터베이스 명령어" 모음
  │   ├─ 📄 WaterDatabase.kt      → "데이터베이스" 설정
  │   └─ 📄 WaterRepository.kt    → "창고 관리자"
  │
  ├─ 📁 domain (2층 - 계산 관련)
  │   └─ 📄 DailyWaterIntake.kt   → "하루 물 섭취량" 계산기
  │
  └─ 📁 presentation (3층 - 화면 관련)
      ├─ 📄 BluetoothManager.kt   → "블루투스 무전기"
      ├─ 📄 HomeViewModel.kt      → "화면 관리자"
      └─ 📄 HomeScreen.kt         → "실제 화면"
```

## 3. 데이터 저장소 (창고)

### 3.1 데이터베이스란? (서류철)

**데이터베이스**는 **서류철**과 같아요!

```
📋 서류철 (데이터베이스)
  │
  ├─ 📄 1월 8일 - 오전 9시: 200ml 마심
  ├─ 📄 1월 8일 - 오후 1시: 300ml 마심
  ├─ 📄 1월 8일 - 오후 3시: 150ml 마심
  └─ 📄 ...
```

각 기록마다 다음 정보를 저장해요:
- **ID** (순서 번호): 1번, 2번, 3번...
- **양** (amount): 200ml, 300ml...
- **시간** (timestamp): 오전 9시 30분...
- **날짜** (date): 2025년 1월 8일

### 3.2 WaterRecord (기록 한 장)

**하나의 기록**이 어떻게 생겼는지 보세요:

```kotlin
// 이건 "물 마신 기록 한 장"을 나타내요
data class WaterRecord(
    val id: Long,           // 순서 번호 (자동으로 1, 2, 3...)
    val amount: Int,        // 마신 양 (ml)
    val timestamp: Long,    // 정확한 시간 (밀리초 단위)
    val date: String        // 날짜 (2025-01-08 형식)
)
```

**예시:**
```
기록 #1:
  - ID: 1
  - 양: 200ml
  - 시간: 2025-01-08 09:30:00
  - 날짜: 2025-01-08
```

### 3.3 데이터 꺼내기 (검색하기)

서류철에서 원하는 자료를 찾는 것처럼, 데이터베이스에서도 검색할 수 있어요:

**질문:** "오늘(1월 8일) 기록을 모두 보여줘!"

```kotlin
// 날짜로 검색하기
fun getRecordsByDate(date: String): Flow<List<WaterRecord>>

// 예시: "2025-01-08"로 검색하면
// 1월 8일에 마신 모든 기록이 나와요
```

**질문:** "오늘 총 얼마나 마셨어?"

```kotlin
// 합계 계산하기
suspend fun getTotalAmountByDate(date: String): Int

// 예시: 200ml + 300ml + 150ml = 650ml
```

### 3.4 Flow가 뭐예요? (실시간 알림)

**Flow**는 **실시간 알림**과 같아요!

**일반 방식 (옛날 방식):**
```
👤 사용자: "지금 얼마나 마셨어?"
📱 앱: "650ml요." (한 번만 대답)

... 나중에 또 물을 마심 ...

👤 사용자: "다시 물어볼게, 지금 얼마나 마셨어?"
📱 앱: "800ml요." (다시 물어봐야 대답함)
```

**Flow 방식 (똑똑한 방식):**
```
👤 사용자: "물 마실 때마다 알려줘!"
📱 앱: "650ml요."

... 자동으로 ...

📱 앱: "800ml로 바뀌었어요!" (자동 알림)
📱 앱: "950ml로 바뀌었어요!" (자동 알림)
```

## 4. 블루투스 통신 (무전기)

### 4.1 블루투스란? (무전기)

**블루투스**는 Arduino와 스마트폰을 연결하는 **무전기**예요!

```
[Arduino 무게 센서] 📡 ↔️ 📱 [스마트폰 앱]
      "250g이야!"  →
      "50ml 마셨어!" →
                  ← "영점 조정해줘!"
```

### 4.2 메시지 형식 (암호)

Arduino와 앱이 주고받는 **비밀 암호** 같은 규칙이 있어요:

| 암호 | 뜻 | 예시 |
|-----|---|-----|
| `W:` | 지금 무게는... | `W:250.5` → "지금 250.5g이야" |
| `D:` | 마신 양은... | `D:50.0` → "50ml 마셨어!" |
| `S:` | 상태는... | `S:READY` → "준비됐어!" |
| `T` | 명령: 영점 조정 | `T` → "0으로 맞춰줘" |

**실제 대화 예시:**
```
Arduino: "W:0.0"        → 무게 0g (빈 받침대)
Arduino: "W:250.5"      → 무게 250.5g (컵 올림)
Arduino: "D:50.0"       → 50ml 마심 감지!
앱: "T"                 → 영점 조정 요청
Arduino: "S:TARE_OK"    → 조정 완료!
```

### 4.3 데이터가 조각나서 올 수 있어요!

무전기를 쓸 때 가끔 **말이 끊기는 것**처럼, 블루투스도 그래요:

**문제 상황:**
```
보내는 메시지: "W:250.5"

실제로 받는 모습:
1번째: "W:25"      ❌ 불완전
2번째: "0.5"       ❌ 불완전
```

**해결 방법: 조각 모으기**
```kotlin
// 조각들을 모아서 완전한 문자를 만들어요
StringBuilder를 사용해서 조각 붙이기

"W:25" + "0.5" = "W:250.5" ✅ 완성!
```

마치 **퍼즐 조각**을 맞추는 것과 같아요!

### 4.4 BluetoothManager (무전기 담당자)

**BluetoothManager**는 무전기를 관리하는 친구예요:

```
BluetoothManager가 하는 일:

1️⃣ 연결하기
   → Arduino와 연결해요

2️⃣ 듣기
   → Arduino가 보낸 메시지를 계속 듣고 있어요

3️⃣ 해석하기
   → "W:250.5" → "250.5g이네!"

4️⃣ 보내기
   → 앱이 보내고 싶은 명령을 Arduino에 전달해요
```

## 5. 화면 그리기 (그림 그리기)

### 5.1 Jetpack Compose란? (레고 블록)

**Jetpack Compose**는 화면을 만드는 **레고 블록** 같아요!

```
🧱 작은 블록들을 조립해서 큰 화면을 만들어요

[버튼] + [글자] + [진행 바] = 완성된 화면!
```

**기존 방식 (복잡):**
- XML 파일에 디자인 그리기
- Java/Kotlin 코드로 조종하기
- 둘을 연결하기 (복잡!)

**Compose 방식 (쉬움):**
- 코드 한 곳에서 모두 해결!
- 레고 블록 쌓듯이 조립!

### 5.2 Composable (레고 블록)

**@Composable**은 **하나의 레고 블록**이에요:

```kotlin
// "버튼" 블록 만들기
@Composable
fun MyButton() {
    Button(onClick = { /* 클릭하면 할 일 */ }) {
        Text("눌러보세요!")
    }
}

// "글자" 블록 만들기
@Composable
fun MyText() {
    Text("안녕하세요!")
}

// 블록들을 조합해서 큰 화면 만들기
@Composable
fun MyScreen() {
    Column {  // 세로로 쌓기
        MyText()
        MyButton()
    }
}
```

**결과:**
```
┌─────────────┐
│ 안녕하세요!  │
│ ┌─────────┐ │
│ │눌러보세요!│ │
│ └─────────┘ │
└─────────────┘
```

### 5.3 State (상태 = 변하는 값)

**State**는 **시계의 시간**처럼 **계속 바뀌는 값**이에요:

```kotlin
// currentWeight는 "현재 무게"를 저장해요
val currentWeight by viewModel.currentWeight.collectAsState()

// 화면에 표시
Text("현재 무게: ${currentWeight}g")
```

**동작 방식:**
```
처음: currentWeight = 0g
  → 화면: "현재 무게: 0g"

무게 센서 감지!
  → currentWeight = 250.5g
  → 화면: "현재 무게: 250.5g" (자동으로 바뀜!)
```

마치 **시계가 자동으로 시간을 업데이트**하는 것처럼,
State가 바뀌면 **화면이 자동으로 다시 그려져요**!

## 6. 자동으로 물 마신 양 기록하기

### 6.1 어떻게 자동으로 알 수 있을까?

Arduino가 **탐정**처럼 무게 변화를 감시해요:

```
1️⃣ 컵 감지
   빈 받침대: 0g
   컵 올림: 250g
   → "컵이 올려졌네!" 감지

2️⃣ 무게 모니터링
   처음: 250g
   조금 후: 250g
   또 조금 후: 250g
   → "아직 안 마셨어"

3️⃣ 무게 감소 감지!
   처음: 250g
   조금 후: 200g
   → "어? 50g 줄었네!" 감지

4️⃣ 확인 (안정화)
   5번 연속으로 200g인지 확인
   → "정말로 50g(ml) 마셨구나!"

5️⃣ 앱에 알림
   Arduino: "D:50.0" 전송
   → 앱: 자동으로 기록!
```

### 6.2 앱이 기록하는 과정

Arduino가 `D:50.0`을 보내면 이런 일이 일어나요:

```
1️⃣ BluetoothManager (무전기 담당)
   → "D:50.0" 받음
   → 해석: "50ml 마셨대!"
   → drinkAmount = 50.0

2️⃣ HomeViewModel (관리자)
   → drinkAmount 변화 감지!
   → "기록해야겠다!"
   → Repository에게 "50ml 저장해줘" 요청

3️⃣ Repository (창고 관리자)
   → 기록 만들기:
     - 양: 50ml
     - 시간: 2025-01-08 14:30
     - 날짜: 2025-01-08
   → 데이터베이스에 저장

4️⃣ Database (서류철)
   → 기록 추가 완료!
   → "새 기록이 추가됐어!" 알림 (Flow)

5️⃣ HomeScreen (화면)
   → Flow 알림 받음
   → "화면 업데이트 해야지!"
   → 650ml → 700ml로 자동 변경!
```

전부 **자동**이에요! 여러분은 그냥 물만 마시면 돼요!

## 10. 자주 묻는 질문 (FAQ)

### Q1: 코드가 너무 많아요. 다 알아야 하나요?

**A:** 아니요! 처음에는 **전체 그림**만 이해하세요.
- 앱이 **3층 구조**로 되어있다
- **데이터베이스**에 기록을 저장한다
- **블루투스**로 Arduino와 통신한다
- **자동으로** 물 마신 양을 기록한다

이 정도만 이해해도 충분해요!

### Q2: Flow가 어려워요. 꼭 알아야 하나요?

**A:** Flow는 **"자동 알림"**이라고만 기억하세요!
```
Flow 없이: "얼마 마셨어?" 물어봐야 대답
Flow 있으면: 변할 때마다 자동으로 알려줌
```

### Q3: ViewModel, Repository는 왜 필요해요?

**A:** 정리 정돈 때문이에요!
```
ViewModel: 화면에 보여줄 것을 준비하는 친구
Repository: 데이터를 저장하고 가져오는 친구
```

각자 맡은 일만 하니까 관리가 쉬워져요!

### Q4: Compose는 XML과 뭐가 달라요?

**A:**
```
XML (옛날 방식):
  디자인 파일 + 코드 파일 = 2개 파일

Compose (새로운 방식):
  디자인 + 코드 = 1개 파일 (레고처럼 조립!)
```

Compose가 더 쉽고 현대적이에요!

### Q5: 코루틴(Coroutine)은 뭐예요?

**A:** **동시에 여러 일을 하는 비서**예요!
```
일반 방식:
  일 1 완료 → 일 2 시작 → 일 3 시작 (느림)

코루틴:
  일 1, 일 2, 일 3 동시에! (빠름)
```
